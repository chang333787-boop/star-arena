## I. 규격서 vs 현재 상태: 어긋남 지점

### 1. **캐릭터 — 거의 준수함 (유지)**

| 항목 | 규격 요구 | 현재 상태 | 어긋남 정도 | 작업 |
|---|---|---|---|---|
| **스타일** | 셀셰이딩 치비 (라인 남보라, 2~2.5등신) | reference/02_basic_motion_master.png 마스터 기반 → slice_assets.py v6로 자동 슬라이싱 → 5프레임(idle/move/attack/hit/return) 저장 | 0 (기준 마스터 이미 준수) | ❌ |
| **시점** | 3/4 오블리크 | 마스터 이미 3/4 오브리크 구성 | 0 | ❌ |
| **접지선** | 공유(발밑 기준선 같음) | 캐릭터별 5프레임이 **같은 캔버스 높이 + 가로중앙 + 발 기준(하단)** 정렬됨(slice_assets.py 라인 238~245) | 0 ✅ | ❌ |
| **톤** | 고채도·고대비(주인공) | 마스터 시트 색감 이미 깨끗함(톤다운 필요 없음) | 0 | ❌ |
| **그림자** | 자체 그림자 없음, 엔진 드로우 | slice_assets.py가 배경 투명화 + `remove_bg`로 그림자 제거(L96~110) | 0 ✅ | ❌ |

**결론**: 캐릭터는 규격 준수. 사람이 새 마스터 시트를 넣으면 slice_assets.py가 자동 처리.

## II. 엔진 쪽에서 손봐야 할 곳

### **2-1. ❌ y-sort 렌더링 순서 — 발밑 Y 기준 정렬 없음**

**현재 코드** (index.html L2897~2900):
```javascript
for(const e of enemies) drawFighter(e);
for(const a of allies) drawFighter(a);
drawFighter(player);
```

**문제**: fighters를 **Y 좌표로 정렬하지 않음** → 두 캐릭터가 겹칠 때 어떤 것이 앞/뒤인지 잘못될 수 있음 (규격서 0장 "y-sort" 요구)

**필요한 작업**:
- 모든 fighters(enemies + allies + player) + 구조물(structures)을 **발밑 Y 기준(f.y 또는 f.y+f.r)**으로 정렬 후 그리기
- 규격서: "y-sort(발밑 Y가 큰 물체를 나중에 그림)"

---

### **2-2. ❌ 공통 그림자 드로우 — fighters 아래 타원 그림자 없음**

**규격서 요구** (0장): "그림에 굽지 않음 → 엔진이 발밑 공통 타원"

**현재 코드**:
- drawStructsPacked(L3356~3370)에서는 구조물 아래 타원 그림자를 그림: `ctx.ellipse(st.x, st.y+def.h*0.42, def.w*0.62, def.h*0.2, ...)`
- **fighters(캐릭터·몬스터)는 그림자가 없음** ← 각 이미지에 자체 그림자가 있다고 가정하는 것 같음

**문제**: 새 에셋들이 "자체 그림자 없음(투명배경)"이므로 엔진이 그려야 함

**필요한 작업**:
- `drawFighter()` 전에 (또는 y-sort 그리기 루프에서) f.y/f.r 기준으로 **공통 타원 그림자** 드로우
- 예: `ctx.fillStyle="rgba(0,0,0,0.20)"; ctx.ellipse(f.x, f.y+f.r*0.9, f.r*0.7, f.r*0.25, 0, 0, 2*Math.PI); ctx.fill();`

---

### **2-3. ⚠️ ASSET_MANIFEST 배선 — 새 파일 포맷 확인 필요**

**현재 상태** (L548~612):
```javascript
const ASSET_MANIFEST = {
  tiles:     { ground_basic: ..., ground_path: ..., capture_zone: ... },
  obstacles: { block_01: ..., crystal_01: ..., cushion_wall: ... },
  bushes:    { bush_01: ..., bush_02: ... },
  ui:        { icon_hp: ..., ... }
};
```

**문제**:
- 현재는 **선언만** 있고 **사용 안 됨**(주석: "내일 확장용")
- 몬스터는 따로 `getMonsterImage()` 함수로 처리(L610~611)
- 타일셋은 따로 `worldImage()` 함수로 처리(L621)

**필요한 작업**:
- Phase 1에서 새 시트 도입 시 **ASSET_MANIFEST 활용 방식** 확정
- 현재는 파일명 직접 배선이라 유지보수 어려움 → 확인 후 리팩터링 고려

---

### **2-4. footprint 데이터 모델 도입 — 없음**

**규격서 1장 표**에서 각 오브젝트의 **footprint(칸 단위)**가 명시됨:
- 캐릭터: 1×1
- 보스: 2×2
- 몬스터 문: 2×1
- 수정·바위: 1×1
- 등등

**현재**: mapData에 `{x,y,w,h}` 픽셀 단위만 있음 → **칸 단위 footprint 없음**

**필요한 작업**:
- GAME_CONFIG나 별도 파일에 **footprint 데이터**: `FOOTPRINTS = { crystal_01: {cols:1, rows:1}, boss: {cols:2, rows:2}, ... }`
- 맵 에디터에서 "이 칸에 배치 가능한가?" 판정할 때 사용

## IV. 슬라이서 파라미터화 — 계획

### **현재 상태**
`tools/slice_assets.py` v6:
- 고정 대상: reference/02_basic_motion_master.png (6행 × 5열)
- 출력: characters/student_01~06 에 5프레임씩
- 특수: reference/01_fixed_character_weapon_master.png → weapons/tool_01~06 아이콘

### **필요한 확장**
새로운 "월드 마스터 시트"(몬스터·타일·장애물·프롭 통합 또는 개별)을 자를 때:
- **파라미터화** 필요: 행·열 수, 셀 크기, 출력 경로 다르면 다른 설정 필요

### **제안 구조**
```python
# tools/slice_assets.py 내 SHEET_CONFIGS 딕셔너리 추가:
SHEET_CONFIGS = {
  "motion": {  # 기존
    "source": "reference/02_basic_motion_master.png",
    "rows": 6, "cols": 5,
    "outDir": "assets/characters",
    "names": ["student_01", ...],
    "frames": ["idle", "move", "attack", "hit", "return"]
  },
  "world": {   # 새로 추가 (아직 없음)
    "source": "reference/04_world_master.png",  # 사람이 만들면 여기에
    "rows": ?, "cols": ?,
    "outDir": "assets/monsters",  # 또는 "assets/obstacles" 등
    ...
  }
}
```

### **구현 포인트**
1. **행/열 자동 감지** vs **고정 지정**: 현재 v6은 헤더 파란 박스로 열 감지(L149~160). 행은 고정(6행). 신규는 이미지마다 다를 수 있으므로 설정으로 해야 함
2. **배경제거 로직 유지**: 신규 시트도 배경투명 + halo 정리 필수
3. **출력 파일명**: 규격에 맞춰 타입 자동 지정 (예: monsters/\<type\>.png)

### **실행 계획**
1. **파이썬 코드 리팩토링**: 현재 하드코딩된 부분을 설정으로 뽑아내기
2. **사람이 새 시트 제공** → 설정 추가 후 실행
3. **QA**: 결과 파일이 규격 준수하는지 테스트
