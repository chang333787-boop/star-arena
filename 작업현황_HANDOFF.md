### (구) 3v3 재설계 조사 기록 (2026-07-07, Phase 1로 해소됨)
- **진단 확정:** 3v3 순간이동·렉의 근본 원인은 넷코드 로직이 아니라 **전송 경로**. 현 구조는 Firebase RTDB(asia-southeast1=싱가포르)를 15Hz 게임 중계로 사용 → 같은 교실 와이파이라도 입력·상태가 매번 한국↔싱가포르 왕복(150~300ms). v1.50/v1.51 넷코드 수정으로는 이 구조적 왕복을 못 없앰.
- **방향 결정(리서치 4갈래 wf_3cb8405b 근거):** 상용 파티게임 방식 = **WebRTC DataChannel P2P**(같은 와이파이 기기끼리 직접 연결, 왕복 5~20ms). 도입 형태는 순수 P2P가 아니라 **"WebRTC 먼저 + 실패 시 현행 RTDB 자동 폴백" 하이브리드**(TURN 없음). 시그널링만 기존 Firebase 재활용. 서버 0·교사 운영 0·단일 index.html 유지. 격리 교실도 최악의 경우 지금과 동일(퇴행 0).
  - 대안 기각: 관리형 서비스(Ably 등)=현행 대비 이득 미미(단 WS차단이 원인이면 Ably가 유일 개선책) / 자체 WS서버=교사 운영 불가+생WS는 폴백 없어 방화벽에 더 나빠짐.
- **게이트(핵심):** WebRTC 본작업(1~2주) 전에 **`nettest.html`로 실제 교실 와이파이에서 P2P 직접 연결 여부부터 실측**. 위험변수 = 학교 와이파이 클라이언트(AP) 격리 — 켜져 있으면 P2P 통째 실패, 사전 예측 불가.
  - 배포: https://chang333787-boop.github.io/star-arena/nettest.html (커밋 e7117e0). 독립 파일, index.html 무관.
  - 사용: **서로 다른 기기 2대**로 그 교실 와이파이에서 열기 → 한쪽 방만들기(4자리)·다른쪽 참가. (같은 기기 2탭은 항상 성공이라 무의미.)
  - 판정: 🟢 직접 LAN/STUN 연결됨 = 본작업 진행 / 🔴 15초 내 실패 = 격리 의심, WebRTC 넣어도 이 교실선 폴백만 씀.
  - 측정: WebRTC 왕복 vs Firebase 왕복 A/B 나란히 표시(로컬 2탭 검증치 1ms vs 160ms).
- **다음 단계:** 교사가 교실 실측 → 🟢면 하이브리드 구현 착수(writeHostState/writeInput/onlineState/onlineInput 4개 전송지점을 DataChannel로 스위치+RTDB 폴백 유지), 🔴면 Ably 폴백 검토.
